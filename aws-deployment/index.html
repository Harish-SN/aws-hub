AWS Deployment Methods

This guide covers AWS deployment methods used for applications, infrastructure, containers, and serverless workloads. These concepts are critical for Solutions Architect and DevOps Professional exams.

All-at-Once Deployment

Type: Basic Deployment Strategy

Overview

Deploys the new version to all targets simultaneously.

  • Fastest deployment
  • High risk of downtime
  • No gradual traffic shifting

Used With: EC2, On-prem servers (CodeDeploy)

Rolling Deployment

Type: Incremental Deployment

Overview

Deploys the application in batches instead of all at once.

  • Reduced downtime
  • Users may see mixed versions
  • Rollback is slower

Used With: EC2 Auto Scaling, ECS

Blue / Green Deployment

Type: Zero-Downtime Deployment

Overview

Two environments exist: Blue (current) and Green (new). Traffic is switched after validation.

  • Instant rollback
  • No downtime
  • Higher cost (duplicate environments)

Used With: EC2, ECS, Lambda, Elastic Beanstalk

Canary Deployment

Type: Progressive Traffic Shift

Overview

Releases the new version to a small percentage of users first.

  • Early bug detection
  • Very low risk
  • Traffic increases gradually

Used With: Lambda, API Gateway, ECS, ALB

Linear Deployment

Type: Gradual Deployment

Overview

Traffic is shifted in equal increments over time.

  • Predictable rollout
  • Safer than rolling
  • Slower than canary

Used With: Lambda (CodeDeploy)

A/B Deployment

Type: Experiment-Based Deployment

Overview

Different user groups receive different versions of the application to compare behavior or performance.

  • Used for experimentation
  • Business-driven decision making
  • Not purely a release strategy

Used With: CloudFront, ALB, feature flags

Shadow Deployment

Type: Testing Deployment

Overview

Production traffic is duplicated and sent to a new version without impacting users.

  • No user impact
  • Excellent for performance testing
  • Complex to implement

Used With: ALB, microservices architectures

Immutable Deployment

Type: Infrastructure Replacement

Overview

New infrastructure is created for each release. Old infrastructure is destroyed after success.

  • No configuration drift
  • Highly reliable
  • Higher infrastructure cost

Used With: EC2 + AMI, Auto Scaling Groups

Recreate Deployment

Type: Stop & Replace

Overview

Old version is stopped before the new version is deployed.

  • Simple
  • Causes downtime
  • Rarely used in production

Feature Flag / Dark Launch

Type: Runtime Deployment Control

Overview

Code is deployed but features are enabled or disabled at runtime.

  • No redeployment required
  • Used for gradual rollouts
  • Often combined with canary or A/B

Used With: AppConfig, custom flags
